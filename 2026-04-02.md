1. Go的channel有/无缓冲的区别：

   无缓冲channel，发送方和接收方必须同时准备好；发送者阻塞直到有接收者，接收者阻塞直到有发送者；直接从发送Goroutine到接收Goroutine；发送成功则意味着接收方已收到。

   有缓冲channel，缓冲区未满，发送方即可发送；缓存区未空，接收方即可接收；缓冲区满时发送阻塞，缓冲区空时接收阻塞；先拷贝到缓冲区，再从缓冲区拷贝到接收者；发送成功仅代表数据已成功入队

2. 如果一个channel已经关闭，继续往里写数据会怎么样，读数据又会怎么样：

   写数据会panic，读数据会返回0值；使用`comma ok`来确认发送的不是0值

3. `defer`的执行顺序：

   先进后出；

   当defer语句被声明时，它后面的参数会被立即计算并固定下来

   ```go
   func deferTrap1() {
       i := 0
       defer fmt.Println("A:", i) // 压栈时 i 已经固定为 0
       
       defer func() {
           fmt.Println("B:", i)   // 闭包引用，执行时才去读取 i 的内存值
       }()
       
       i++
   }
   // 打印顺序：
   // B: 1
   // A: 0
   ```

4. 读写锁(RWMutex)和互斥锁(Mutex)的区别：

   都是用于解决数据并发安全的，核心区别在于对并发的包容度；

   互斥锁，一次只有一个Goroutine能拿到锁，写操作期间阻塞任何读操作

   读写锁，多个读操作可以 同时进行，且写操作期间也阻塞任何读操作

   一个Goroutine在未释放锁的情况下申请另一个锁会触发死锁和panic

5. 什么场景下用 Mutex，什么场景下用 RWMutex？

   **读多写少**的场景用Mutex，如缓存查询，字典查询

   **写多读少**用RWMutex

6. Go 的 map 是线程安全？不安全，怎么让多个线程访问安全？（加锁 读写锁互斥锁）:

   Go map 原生不安全，因为设计者认为大多数情况下map不需要并发访问；如果map检测到多个GoRoutine同时访问map会触发报错

   可以采用`sync.Map`，前提是写操作较少，读操作较多的前提下

7. Go 协程的调度怎么实现，GMP 模型

   GMP:

   + G代表Goroutine，是go代码执行的实体，内部保存了该协程的栈信息、状态、指针等；它的执行依赖于M
   + M代表Machine，操作系统的层面的内核线程，负责执行G中的代码
   + P代表Processor，包含了运行G所需的资源和一个本地可运行G队列

   除GMP外还有一个全局队列(Global Queue)，当P的队列满时会将G放入其中

   + 若M空闲：

     若分给M的G执行完毕且P中空闲，则会试图去全局队列和其它P中偷取G进行执行

   + 若M被阻塞：

     P会主动与当前的M解绑，然后去寻找其它空闲M或申请一个新M来执行G；之前阻塞的M在醒来后会主动寻找空闲的P绑定，若无法找到则把执行完的G扔到全局队列，然后自己进入休眠池。

   + 如果一个 G 是一个没有 I/O 操作的纯 CPU 密集型死循环，它一直占用 M，其他 G 岂不是永远得不到执行？

     Go运行时会有一个监控线程(`sysmon`)来周期性地检查G，如果某个线程执行超过10ms，该进程就会给其打上标记，下一次执行时就会强制剥夺并将其放到全局队列，以此来让其它进程执行。

   + 为什么要有P？

     只有GM的话，所有GM都要去竞争全局队列，会导致效率低下

8. HTTP请求流程：

   寻址，建连，交互，断连：

   + 首先解析URL，通过DNS协议将域名解析为IP
   + TCP建立连接，拿到目标IP后，客户端的传输层回想服务端的80端口(http)或443端口(https)发起TCP连接；如果是https，还需要TLS握手来协商加密算法和交换对称密钥
   + 浏览器构建 HTTP 请求报文（包含请求行、请求头、请求体）。数据沿着网络模型往下传，最后发给服务器。
   + 请求到达服务器后，通常先经过**反向代理/网关**（如 Nginx），然后被转发到**后端应用**
   + 四次挥手断连
